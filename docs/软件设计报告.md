# 校园水浸监测系统 - 软件设计报告

## 1. 引言

### 1.1 编写目的
本说明书旨在全面描述“校园水浸监测系统”的软件架构、功能模块、数据结构、接口规范及部署策略。作为开发、测试和维护的基线文档，以指导后续系统的迭代优化及日常运维工作。

### 1.2 系统定位
校园水浸监测系统是一套面向园区、学校和地下空间的 IoT 防汛监测平台。主要用于接入超声波水位传感器和浸水传感器，提供稳定可靠的数据采集、地图展示、阈值告警、通知推送以及设备运维管理功能。

## 2. 总体架构设计

### 2.1 技术选型
*   **设备接入层**：EMQX (MQTT Broker) + Paho-MQTT
*   **API与业务层**：Python + FastAPI + Pydantic + SQLAlchemy Async
*   **数据持久层**：PostgreSQL 16
*   **前端展示层**：Python + Streamlit
*   **部署交付**：Docker + Docker Compose

### 2.2 逻辑架构
系统按功能抽象分为四层结构：
1.  **接入层**：EMQX 负责保持与现场硬件设备的大量并发连接，接收 MQTT 报文。FastAPI 后端内部启动 MQTT 守护线程进行数据消费与兼容性转换。
2.  **业务层**：传感器管理、读数解析入库、告警规则引擎计算核心业务。
3.  **数据层**：PostgreSQL 存储设备的档案信息、实时传感状态、序列化读数历史和系统操作告警。
4.  **展示层**：基于 Streamlit 动态直出仪表盘和空间水位地图。

### 2.3 部署拓扑
```text
[ 现场传感器节点 ]
       | (MQTT/TCP)
       v
[ EMQX Broker (:1883) ]
       | (paho-mqtt)
       v
[ FastAPI 后端 (:8000) ] <---> [ PostgreSQL ]
       ^
       | (HTTP/REST API)
[ Streamlit 运维界面 (:8501) ] 
```

## 3. 核心功能模块设计

### 3.1 接入与解析模块
*   **MQTT 报文处理**：通过订阅 `hbbwater/{device_type}/{device_id}/data` 接收数据。
*   **自动注册机制**：对未在系统登记的合法类型设备，基于默认策略（如基础阈值、默认坐标等）自动落库，使得硬件通电即可在云端显示。
*   **异构兼容解析**：兼容多种传感器的数据字典，将 `mm` 统一转换为 `cm`，布尔值统一转换为 `state`。

### 3.2 告警规则引擎
对每次新到站的数据进行并发评估：
1.  **超声波水位规则**：对比读数与 `threshold_warn` (预警阈值) 以及 `threshold_danger` (危险阈值)。若 `threshold_dir` 为 `greater_or_equal`，高于判断线即触发相应告警（预警 `high_water` 或危险级别）。
2.  **浸水传感规则**：仅判断是否产生水浸 (`water_detected=true` 或 `value>=1`)，若触发则生成 `water_detected` 级危急告警。
3.  **抑制与恢复**：加入告警冷却时长 (`ALERT_COOLDOWN_MINUTES`) 防止告警风暴；若设备状态回归正常，则将告警自动标为 `resolved`。

### 3.3 地图与仪表盘
*   **地理空间分布**：将点位叠加至校园底图（支持自定义更换地图资源）。
*   **在线状态跟踪**：结合离线超时 (`OFFLINE_TIMEOUT_SECONDS`)，对超时未上报的设备实施离线标记。

## 4. 数据模型设计

系统通过 SQLAlchemy 进行关系型数据映射，主要表结构如下：

| 表名 (Table) | 核心字段说明 | 业务职责 |
| :--- | :--- | :--- |
| `users` | `username`, `password_hash`, `role` | 存储后台控制台管理用户，支持 RBAC。 |
| `sensors` | `device_id`, `type`, `map_x`, `map_y`, `threshold_warn`, `threshold_danger`, `last_value` | 存储设备元数据，含设备ID、GIS坐标及判断阈值，保存快照以加快列表查询。 |
| `sensor_readings` | `sensor_id`, `value`, `battery`, `rssi`, `created_at` | 数据时序明细表。附带复合索引以支撑历史曲线分析。 |
| `alerts` | `sensor_id`, `severity`, `status`, `triggered_at`, `cooldown_until` | 生命周期化的告警记录。 |
| `system_config` | `key`, `value` | kv型动态配置系统。 |

## 5. 接口设计 (API)

### 5.1 数据接入接口
除 MQTT 通道外，预留 HTTP 通用补充接口 (`POST /api/ingest/http`)，便于测试平台或无 MQTT 能力网关的对接。
*   **Payload**：`{"device_id": "...", "type": "ultrasonic", "value": 18.2, "unit": "cm", "battery": 82}`

### 5.2 业务控制 API (RESTful)
供前端或第三方应用访问，默认基于 JWT 鉴权：
*   `/api/sensors`：设备的 CRUD 管理。
*   `/api/sensors/{sensor_id}/readings`：分页拉取设备历史报文数据。
*   `/api/alerts`：实时或历史告警获取，结合 `/api/alerts/{id}/resolve` 手工消除误报。
*   `/api/dashboard/stats` 与 `/api/dashboard/sensor-status`：高频刷新接口，供数据大屏和地图直出渲染。

## 6. 安全与部署设计

### 6.1 安全设计
*   **网络隔离**：数据库 PostgreSQL 及业务内网交互不对外暴露端口。
*   **身份验证**：REST OpenAPI 提供强 JWT 鉴权，密码在库中密文 (Hash) 存储。
*   **容灾隔离**：对原始传感器报文 (Raw JSON) 进行保留入库，便于历史问题的溯源与重放。

### 6.2 部署策略
*   采用 `docker-compose.yml` 整装发布，内含 `frontend`, `backend`, `emqx`, `postgres` 四个容器。
*   **配置管理**：所有环境变量敏感信息 (`JWT_SECRET`, `POSTGRES_PASSWORD`, `WEBHOOK_URL`) 基于宿主机的 `.env` 文件独立注入容器内部。
*   **平滑回档**：升级可通过修改 Tag + 原地重启服务进行，配置维护接口 `/api/config/maintenance` 辅助大体量数据的归档规划。
